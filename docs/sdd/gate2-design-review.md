## 2. 指摘事項

### C-01: POST /api/generate のレスポンス設計矛盾（Critical）

**パターン**: DR-G2-02（SDD内部矛盾）

**問題**: 外部設計書（04）で POST `/api/generate` のレスポンス（202 Accepted）に `images` 配列と `executionTime` が含まれている。しかし以下と矛盾する:

1. **NFR-001**（01-requirements.md）: 「生成キュー投入レスポンス < 500ms」— 画像生成は数秒〜数十秒かかるため、画像を含むレスポンスを 500ms 以内に返すことは不可能
2. **内部設計書（05）**: 「Next.js Route Handler は即 202 を返す」と記載
3. **画面仕様書（02）状態遷移**: `idle → queued → processing → complete` の4段階遷移を定義しており、非同期前提の設計

**現状の矛盾**: 同期的に画像を返す設計（04のレスポンススキーマ）と、非同期でSSE進捗を配信する設計（02/05）が共存している。

**修正案**: 以下のいずれかに統一すること:
- **案A（非同期・推奨）**: POST `/api/generate` は即座に `{ generationId, promptId }` のみを返す（202）。画像URLはSSE完了イベントまたはGET `/api/generations/[id]` で取得
- **案B（同期）**: POST `/api/generate` が画像生成完了まで待機する設計に統一。この場合 NFR-001 の基準値を変更し、SSE進捗エンドポイントは不要

**パターン**: DR-G2-02（SDD内部矛盾）

### W-01: ギャラリー検索・日付フィルターのAPI定義欠落（Warning）

**パターン**: DR-G2-02（FR と UI/UX 仕様の整合性）

**問題**: 画面仕様書（02）のギャラリー画面には以下のUI要素がある:
- 検索ボックス（「プロンプトで検索」、300msデバウンス）
- 日付フィルター（「すべて」「今日」「今週」「今月」）

しかし外部設計書（04）の GET `/api/generations` クエリパラメータには:
- `search` / `q` パラメータが未定義（検索ボックスに対応するAPIがない）
- `dateFrom` / `dateTo` / `dateRange` パラメータが未定義（日付フィルターに対応するAPIがない）

**修正案**: GET `/api/generations` に以下を追加:
```
| search   | string | — | プロンプトテキスト検索（ILIKE） |
| dateRange | string | — | "today" | "week" | "month" |
```

## 7. 総合評価

### 良い点
- デザイントークンが体系的に定義されており、ブレークポイント値の一貫性が保たれている
- セキュリティ設計（入力値サニタイズ、パストラバーサル防止、SQLインジェクション対策）が充実
- V字モデルに基づくテスト戦略が明確で、実DB必須方針が記載されている
- ComfyUI フォールバック（オフライン時）の設計が全レイヤーで考慮されている
- アクセシビリティ要件が画面仕様に含まれている

### 改善すべき点
- **設計ドキュメント間の整合性**: 外部設計とテスト仕様の間にフィールド名・エンドポイント名の乖離が多い。設計フェーズ統合時のクロスチェックが不足している
- **同期/非同期の設計方針**: 画像生成APIの根本的な振る舞いが未確定のまま設計が進んでおり、実装フェーズで大きな手戻りリスクがある
- **Could 要件のスコープ管理**: FR-010（お気に入り）が Phase 1 スコープ外にもかかわらず、テスト仕様に含まれている。スコープ境界を明確にすべき

### Gate 2 判定理由
C-01（同期/非同期矛盾）は実装アーキテクチャの根幹に関わるため、修正なしに実装フェーズに進むと手戻りが大きい。C-02（テスト仕様の不一致）は QA Engineer が外部設計書に合わせて修正すれば解決可能。Critical 2件の修正を条件として **CONDITIONAL** 判定とする。
